## Core Principles

### I. マイグレーション・ファースト（Migration-First）— 最重要原則

このプロジェクトは「新規開発」ではなく「稼働中レガシーシステムの移行」である。
現行仕様の完全踏襲を最優先とし、改善・拡張は段階的に行う。

- **現行仕様は正（Single Source of Truth）**: 01〜06 の分析ドキュメントに記載された
  現行仕様（ビジネスルール、ステータス遷移、権限モデル、バリデーション等）は
  新システムでも 100% 踏襲する。変更する場合は差異管理表（05_gap_analysis.md）に
  差異 ID / 区分 / 現行仕様 / 新仕様 / 変更理由 / 影響範囲 のフォーマットで記録し、
  業務部門の承認を必須とする
- **踏襲必須ビジネスロジック**: 05_gap_analysis.md セクション 4 に定義されたルール群は変更禁止:
  - 時間工数制約 7 件（サービス時間 6:00-23:30、15分単位、日次24h上限）
  - ステータス制御 5 件（12状態マトリクス 000〜911、STATUS_0→1→2 遷移）
  - 入力バリデーション 15 件（必須項目、日付形式、件名128文字上限）
  - 年度期間ルール 3 件（2015年特例3ヶ月、2016年以降6ヶ月）
  - 禁止語句 12 語（件名フィールド）
  これらはテストで 100% カバーすること
- **12状態ステータスマトリクス**: `sts_base_key`（000〜911）の12状態 ×
  担当者/管理者系列のボタン制御は完全再現必須。
  全パターンのパラメタライズドテストを実装すること
- **エラーメッセージ体系の保全**: CZ-000〜CZ-999 のエラーコード体系（92件）は踏襲。
  新システムの API レスポンスにも同一エラーコードを使用する
  - CZ-000〜099: 成功メッセージ
  - CZ-100〜299: 警告メッセージ
  - CZ-300〜499: システムエラー
  - CZ-500〜799: 確認ダイアログ
  - CZ-800〜999: 情報メッセージ
- **SPA 改善提案は段階的に**: 05_gap_analysis.md の ADD 項目（ENH-001〜008:
  オートセーブ、非同期バリデーション、ダッシュボード等）は Phase 4 以降で実装。
  Phase 1〜3 は既存機能の移行に集中する
- **spec-kit 各機能の spec.md 作成時**: 03_user_stories.md のユーザーストーリーと
  ビジネスルールを正として参照し、仕様の出典を明記すること

### II. 仕様完全性（Spec Completeness）

仕様書は実装の唯一の情報源であり、
実装に必要な情報が不足した仕様は手戻りとコストの原因になる。
すべての機能仕様（spec.md）は、以下のチェックリストを満たしてから
次のフェーズ（plan）に進むこと。

#### 画面・UI 成果物

- [ ] **ワイヤーフレーム**: 画面レイアウトの概略図。
      04_screen_transition.md の SPA 統合提案に準拠した配置であること
- [ ] **画面インタラクション設計**: ボタンクリック・フォーム入力・
      ホバー等の挙動を明記。「何をしたら何が起きるか」が曖昧なまま実装に入らない
- [ ] **画面遷移図**: 正常系・エラー時・キャンセル時の遷移先を含む
- [ ] **状態表示パターン**: ローディング / データなし / エラー / 成功 の4状態を定義

#### データ・API 成果物

- [ ] **データモデル**: エンティティ定義とリレーション（Oracle→PostgreSQL 変換済み）
- [ ] **API エンドポイント仕様**: URL、HTTPメソッド、リクエスト/レスポンス、ステータスコード
- [ ] **バリデーションルール**: 03_user_stories.md の VR-001〜VR-015 を参照し、
      各入力項目の許容値・桁数・必須/任意・形式チェックを一覧化

#### 品質・テスト成果物

- [ ] **受け入れ基準**: BDD 形式（Given-When-Then）で検証可能な条件を記述
- [ ] **エラーハンドリング方針**: CZ エラーコード体系に基づくエラーケースと通知方法
- [ ] **エッジケース・境界値**: 上限値・下限値・空入力・同時操作等の境界条件

#### マイグレーション固有

- [ ] **現行仕様との対応表**: 03_user_stories.md の対象 US-ID を明記
- [ ] **差異一覧**: 05_gap_analysis.md の該当 GAP-ID を参照し、変更点を明示
- [ ] **踏襲ビジネスルールの明示**: 該当する BR / VR ルールを列挙

### III. Docker-First Development（Docker中心開発）

すべての開発・テスト・実行は Docker コンテナ上で行う。
ローカル環境依存を排除し、チーム全体で完全に再現可能な開発環境を保証する。

- すべてのサービスは `docker-compose.yml` で定義し、
  `docker compose up` のみで開発環境が完全に立ち上がる状態を維持する
- 環境変数は `.env` ファイルで一元管理し、
  ハードコードされた設定値をコードベースに含めてはならない
- マルチステージビルドを活用し、開発用と本番用のイメージを
  同一 Dockerfile 内で管理する
- ヘルスチェックによるサービス間の依存制御を必須とし、
  起動順序の暗黙的な仮定に依存してはならない
- 名前付きボリュームでデータ永続化とビルドキャッシュを管理する

**開発環境構成（06_devenv_infrastructure.md 準拠）**:

| コンテナ | イメージ | ポート | 役割 |
|---------|---------|-------|------|
| frontend | node:22-slim | 3000 | Nuxt.js 3 SPA（HMR 対応） |
| backend | eclipse-temurin:21-jdk | 8080 | Spring Boot REST API |
| batch | eclipse-temurin:21-jdk | — | Spring Batch（オンデマンド） |
| auth-mock | node:22-slim | 8180 | ALB+Okta OIDC 模倣サーバー |
| db | postgres:16 | 5432 | PostgreSQL（初期化 SQL 自動実行） |
| redis | redis:7-alpine | 6379 | パラメータキャッシュ |

### IV. テスト駆動開発 — TDD（非交渉事項）

t_wada（和田卓人）のテスト駆動開発理論に基づき、
すべての機能実装はテストファーストで進める。

- **Red-Green-Refactor サイクルを厳格に遵守する**:
  1. **Red**: 失敗するテストを先に書く
  2. **Green**: テストを通す最小限の実装を行う
  3. **Refactor**: テストが通る状態を維持しながらコードを改善する
- テストなしのコードをメインブランチにマージしてはならない
- テストは「動作する仕様書」として扱い、可読性・保守性も重視する
- テストの種類と責務を分離する:
  - **単体テスト**: 個々のクラス・関数の振る舞いを検証
  - **統合テスト**: サービス間・DB 連携の動作を検証
  - **E2E テスト**: ユーザーシナリオ全体の動作を検証

#### マイグレーション固有のテスト要件

- **現行仕様の回帰テスト**: 各機能移行時、03_user_stories.md の
  ユーザーストーリーに対応する E2E テストを先に作成
- **ステータスマトリクステスト**: 12状態 × 2系列の全パターンをパラメタライズドテストで網羅
- **バリデーションルールテスト**: 05_gap_analysis.md セクション 4 の全ルールをテスト化
- **エラーメッセージテスト**: CZ-000〜CZ-999 の発生条件をテストで再現

**カバレッジ目標**:
- 新規コード: 80% 以上のラインカバレッジ
- 踏襲必須ビジネスロジック（ステータスマトリクス、バリデーション、工数計算）: **100%**

### V. ユーザー体験最優先（UX-First）

04_screen_transition.md セクション 7 の SPA 統合提案に準拠し、
ユーザー体験を最優先に設計する。

- **SPA 統合**: 18画面 + 10ダイアログ → タブ + モーダルベースの SPA に統合
  - FORM_030 ↔ FORM_040 のタブ切り替え
  - 3段階ドリルダウン（分類別 → システム別 → サブシステム別）
  - window.open() → モーダルダイアログ（8種）
  - パンくずナビゲーション
- **88 Unit → 約20 REST API** への集約
- UI コンポーネントは PrimeVue を使用し、再利用可能な単位で設計する
- レスポンシブデザインを標準とし、デスクトップファーストで快適な操作を保証する
- ユーザー操作に対するフィードバック（ローディング、エラー表示、成功通知）を必ず実装する
- アクセシビリティ（WCAG 2.1 Level A）を遵守し、
  セマンティック HTML と適切な ARIA 属性を使用する
- **デザインスタイル: フラット & ソフトカラー基調**
  - 純粋な黒 (#000) を使わない: テキスト色には Slate-700 (`#334155`) を標準とする
  - フラット & ボーダー基調: コンポーネントの区切りには `elevation="0"` + `border` を使用
  - トナールボタン: 標準ボタンは控えめなスタイル。主要アクションのみカラー指定で区別

### VI. Production Safety — No Dev Feature in Prod

開発時の利便性機能（認証モック、アクター切替 UI 等）が
本番環境に混入し動作することを一切許容しない。

06_devenv_infrastructure.md セクション 5 に定義された三重安全策に準拠:

1. **環境変数ゲート**: `NUXT_PUBLIC_ENABLE_ACTOR_SWITCH` による条件付きビルド
2. **ビルド時 Tree Shaking**: Nuxt.js ビルド時にアクター切替コードを除外
3. **CI パイプライン検査**: ci-production.yml での grep 検査（actor-switch キーワード検出）

- **守りと攻めの対称的テスト**:
  - **守り**: prod ビルドで dev 機能が無効なことを検証（ci-production.yml）
  - **攻め**: dev ビルドで dev 機能が正常動作することを検証（ci-staging.yml）

### VIII. CI/CD Safety — 2系統パイプライン

06_devenv_infrastructure.md セクション 6 に定義された CI 構成に準拠。

| パイプライン | トリガー | ENABLE_ACTOR_SWITCH | 検証内容 |
|------------|---------|---------------------|---------|
| ci-production.yml | push to main/release | **未設定** | grep 混入検査 + ECR Push |
| ci-staging.yml | PR/push to develop | `true` | アクター切替込み E2E テスト |

- CI ランナーは `ubuntu-latest` を標準とする
- スクリプト実行権限は Git で管理、`.gitattributes` で行末コード統一
- 単体テストは H2 In-Memory DB を使用し、PostgreSQL 不要の高速実行を保証
- PostgreSQL 依存のテストは Runtime Check（別ジョブ）に限定
- Gradle Toolchain 自動解決を設定し、ローカル JDK 不要とする

### IX. 最適技術選定（Pragmatic Technology）

最新のベストプラクティスを取り入れながら、
チーム全体が効率的に開発できる技術を選定する。

- 技術選定は**チームの生産性**と**長期的な保守性**のバランスで判断する
- 新しいライブラリ・フレームワークの導入基準:
  - アクティブにメンテナンスされている（直近6ヶ月以内の更新）
  - 十分なドキュメントとコミュニティが存在する
  - プロジェクトの既存技術スタックと矛盾しない
- 「動くから」ではなく「なぜこの技術を選ぶのか」を明文化してから導入する
- 過度な抽象化・過剰設計を避け、YAGNI 原則を遵守する

---

### X. コード品質・保守性（Sustainable Codebase）

チーム全体でコード品質を維持し、長期的に保守可能なコードベースを構築する。

- すべてのコードは PR 経由でレビューを受け、最低1名の承認を得てからマージ
- 命名規則・コーディングスタイルはプロジェクト全体で統一:
  - Backend: Checkstyle / SpotBugs
  - Frontend: ESLint + Prettier
- コミットメッセージは Conventional Commits 形式に従う
  （`feat:`, `fix:`, `docs:`, `refactor:`）
- 技術的負債は意図的に管理し、発生時は Issue として記録して計画的に解消する

## 開発ワークフロー

すべての機能開発は以下のフローに従う。
マイグレーション固有のステップ（現行仕様参照）を各段階に組み込む。

1. **仕様策定** (`/speckit.specify`)
   - 03_user_stories.md から対象 US を引用
   - 現行ソースコードの解読結果を反映
   - テスト可能な要件として記述

2. **仕様明確化** (`/speckit.clarify`)
   - 曖昧な要件を特定し、現行コードとの照合で明確化
   - コードに表れない暗黙の業務ルールをユーザーから引き出す

3. **実装計画** (`/speckit.plan`)
   - 技術的アプローチを決定
   - 05_gap_analysis.md の該当 GAP-ID を参照
   - Constitution Check を実施

4. **仕様品質検証** (`/speckit.checklist`)
   - 原則 II のチェックリストで検証

5. **タスク分解** (`/speckit.tasks`)
   - テストタスクを実装タスクの前に配置（TDD）
   - 現行仕様の回帰テストを最優先

6. **整合性分析** (`/speckit.analyze`)
   - spec.md・plan.md・tasks.md の横断的な整合性チェック
   - Constitution 準拠チェック含む

7. **実装** (`/speckit.implement`)
   - Red-Green-Refactor サイクル
   - Docker 環境上で動作確認

8. **レビュー・マージ**
   - PR でコードレビュー
   - CI でテスト・リントが通ることを確認

すべてのコミットは以下を満たす必要がある:

- テストケースが新機能をカバーしている（カバレッジ 80%+、踏襲ロジックは 100%）

## Governance

- 本 Constitution はプロジェクトのすべての開発活動に優先する
- 01〜06 の分析ドキュメントは Constitution と同等の拘束力を持つ参照文書とする
- 原則の修正には以下を必須とする:
  - 変更理由の明文化
  - チームメンバーの合意
  - 影響を受ける既存コードのマイグレーション計画
- すべての PR・レビューは本 Constitution への準拠を検証しなければならない
- バージョニングはセマンティックバージョニングに従う:
  - MAJOR: 原則の削除・根本的な変更
  - MINOR: 新しい原則・セクションの追加
  - PATCH: 文言の修正・明確化
